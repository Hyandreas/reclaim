# Judge 2 — Round 1 Review: Driftline

## Total: 13.5 / 15

Driftline is a token-vs-code semantic diff engine that classifies every discrepancy into a closed four-verdict taxonomy (accidental regression / intentional redesign / platform constraint / orphaned token), grounds each verdict in deterministic evidence, and proposes bidirectional fixes — patch the code, or draft a token-change proposal aimed at the designer. It is exposed as an MCP server that Cursor calls natively. This is one of the most thorough, self-aware, and demo-ready proposals I could imagine for this track. It is held back from the very top only by a core concept that is (by the team's own admission) the brief's named example, and by an ambition-to-time ratio that is high even for a strong team.

---

## Novelty — 4.0 / 5

The proposal is refreshingly honest that its scaffold — "a token-vs-code diff engine with a redesign/regression/platform-constraint taxonomy" — is close to an example direction the brief hands you (§5 opens by saying the quiet part out loud). That honesty is welcome, but it does mean the foundation is not itself the novelty. What lifts this well above "solid but expected" are three genuine insights layered on top: (1) **verdicts that flip on screen** — Layer 1's naive distance-based read gets overturned by Layer 2 reasoning over corroborating signals, which is a sharp, demonstrable articulation of "reasoning vs. classification" and is arguably the most memorable beat in the whole demo; (2) **bidirectional reconciliation** — when the evidence shows code moved on purpose, Driftline drafts a *token-change proposal for the designer* rather than patching code, a real reframe that directly targets the under-served "keeping designers and engineers aligned" half of the brief; and (3) **recurrence-as-vote** — "the same off-token value in three unrelated components is the codebase voting for a token that doesn't exist yet," a genuinely insightful and memorable framing that turns `Promote to token` from a button into a system-volunteered recommendation. These are strong. What keeps it at 4.0 rather than higher is that the "taste" layer, examined closely, is largely deterministic-signal-driven (migration ratio ≥ 80%, recurrence ≥ 3, commit corroboration) with the LLM acting as a bounded classifier over those signals. That is a smart, demo-safe engineering choice, but a skeptic can fairly say the "taste" is more a rules-plus-evidence engine than aesthetic judgment — and the brief's hardest question ("enough TASTE to know when something is wrong") is answered somewhat mechanically.

## Implementation Plan — 4.5 / 5

This is the strongest axis and close to exemplary. The stack is specific and appropriate end-to-end (TypeScript, Vite/React/Tailwind, PostCSS + `ts-morph` for real AST walking rather than regex, `culori` for CIELAB ΔE and WCAG math, `better-sqlite3`, `@anthropic-ai/sdk`, `@modelcontextprotocol/sdk`). The two-layer architecture — deterministic detection with zero LLM, then a bounded LLM reasoning layer touching only classification and fix-copy — is exactly right: it makes the demo reproducible, makes the "taste" output testable against a closed enum, and keeps the model from ever inventing a delta or escaping a verified byte range (the "a hallucinated patch is structurally impossible" claim is well-earned by construction). The agentic surface is real and concrete: six individually callable tools (`scan_tokens → scan_repo → diff → classify → propose_fix → apply_fix`) driven by both an internal control loop and an MCP server registered in `.cursor/mcp.json` — a direct, credible honor of the Cursor track rather than a bolt-on. Risk handling is genuinely mature: the Tailwind-resolution step is correctly identified as the schedule-eater and gated at midday Day 1 against real arbitrary-value cases before anything is built on top; dynamic classNames get an explicit `needs-manual-review` fallback declared out of scope rather than discovered on stage; the deterministic core alone is a demoable fallback if the LLM layer wobbles; and §6 now specifies the previously-vague literal-to-token resolution algorithm (theme reverse-map + arbitrary-value ΔE matching + tagged pseudo-class variants) in real detail. The real-vs-mocked line (§7) is honest (cached Figma snapshot, proposal-artifact-only for `update_source`, no multi-repo/auth/CV). Two things hold it at 4.5. First, the breadth claimed as "real" for a ~2-day build is high — six tools, *two* fix directions, git commit-corroboration (the fiddliest signal, requiring a hand-seeded realistic commit history), recurrence aggregation, interactive-state WCAG checks, an audit trail, a dashboard, *and* an MCP server — and while each layer is well-sequenced with a fallback, the aggregate is a lot of surface area. Second, the exact Anthropic API shapes cited (`thinking: { type: "adaptive" }`, `output_config.format`, `strict: true`) should be re-confirmed against the live SDK; the model id `claude-opus-4-8` is real and current, and a wrong param name would only be minor rework, but it is worth verifying before the clock starts rather than mid-build.

## Usefulness / Relevance — 5.0 / 5

This is dead-on the track statement and clearly valuable to a real buyer. It hits every clause: "detecting drift" (Layer 1), "proposing reconciliation" (verdicts + fixes), "reasons about consistency across a product's visual **and interactive** surface" (interaction-state focus/hover/disabled contrast runs through the same pipeline against WCAG 2.1 SC 1.4.11's 3:1 non-text minimum — first-class, not bolted on), and — the clause most proposals under-serve — "keeping designers and engineers aligned without a synchronization meeting," which the bidirectional token-change proposal makes *literal* rather than aspirational, since a flipped verdict produces an actual designer-facing artifact. The buyer is specific and real (platform/design-systems engineer at a 50–1,000-person org who owns the token pipeline), the adoption story is frictionless ("a CI gate, not a workflow change"), and the WCAG-contrast-to-ADA-legal-exposure framing converts "aesthetics" into budgeted risk mitigation — a genuinely compelling value argument. The close-out metric ("14 inconsistencies surfaced, 3 auto-fixed forward, 1 flipped and fixed backward, ~40 min of design QA saved on one PR") is concrete and role-aware. The only faint caveat — and it does not cost a point — is that the whole pipeline assumes a clean W3C Design Tokens JSON export, which many real orgs' messy Figma files don't have; but for the stated buyer who owns the pipeline, that assumption is fair.

## Strengths (concrete)

- **Deterministic-core / bounded-LLM split** is the right architecture for both correctness and demo safety: the model can't invent deltas, can't escape verified byte ranges, and emits a closed-enum verdict — so "taste" is testable, not vibes.
- **The verdict-flip-on-screen** (0:17–0:32) is a genuinely convincing, memorable demo device that shows reasoning-over-evidence instead of a lookup table keyed on which folder a file lives in.
- **Bidirectional fixes** are the real differentiator and the concrete answer to the brief's "alignment" clause: a token-change proposal is a designer-facing artifact, not just another engineer-facing CI check.
- **Interactive-surface coverage is first-class** — pseudo-class variants are tagged in the scanner and fed through real WCAG non-text-contrast math, and there's a dedicated on-camera beat for it (0:32–0:40).
- **Anticipates judge skepticism head-on**: the live terminal `git log` cut to the real commit range (0:40–0:48) proves the reasoning isn't a hardcoded string, and "point at any of the ~14 findings" invites live inspection. Unusually savvy.
- **Honest, hard-defended scope line** (§7) plus a per-milestone demoable fallback and a midday Day-1 gate on the single riskiest step (Tailwind resolution).
- **Recurrence-as-vote** auto-surfaces a "Candidate New Token," a pure-code aggregation that turns a manual escape hatch into a data-driven recommendation.

## Weaknesses / Gaps

- **The core taxonomy is the brief's own example**, and the team admits it. The novel additions are strong but are refinements layered on a scaffold the brief handed over.
- **"Taste" is ultimately threshold-driven** (ratio ≥ 80%, recurrence ≥ 3, commit corroboration). A skeptic can argue this is a rules engine with an LLM narrator — the exact hard question the brief poses.
- **High ambition-to-time ratio**: six real tools, two fix directions, git commit-corroboration on a hand-seeded history, MCP server, dashboard, and audit trail is a great deal to land in two days even with strong sequencing.
- **Git commit-corroboration is the riskiest deterministic signal** — it depends on a carefully seeded, realistic commit history where token and code changes land in specific adjacent commits; if that seeding is sloppy, the flip case (the marquee beat) is undermined.
- **API-surface verification**: the exact `thinking` / `output_config.format` / `strict` param shapes should be confirmed against the live SDK before build; the model id itself is fine.
- **Clean-token-export assumption** is a real adoption friction lightly glossed for orgs whose Figma isn't tidy.

## Prioritized Fixes (to raise the score)

1. **Push novelty past the brief's scaffold (targets Novelty 4.0 → 4.5+).** Give the "taste" layer at least one job the deterministic signals demonstrably *cannot* do — e.g., a case where migration ratio and commit corroboration are ambiguous or *conflict*, and the classifier must reason to a defensible verdict the thresholds alone can't reach. Showing the model resolve a genuinely ambiguous case (not just confirm a ≥80% ratio) is what converts "rules + narrator" into "taste."
2. **De-risk the git commit-corroboration signal explicitly (targets Implementation 4.5 → 5.0).** Add a line to §7 confirming the seeded repo's commit history is authored on Day 1 with the exact token-change-adjacent-to-code-change commits the flip case needs, and state a fallback (e.g., migration-ratio + recurrence alone can carry the flip verdict if blame parsing misbehaves) so the marquee beat can't be sunk by one flaky signal.
3. **Verify the Anthropic API shapes before the clock starts (cheap, protects Implementation).** Confirm `thinking` / `output_config.format` / `strict` param names against the current SDK and note the confirmation; a wrong param name discovered mid-build burns time budgeted elsewhere.
4. **State a cut-order to protect the two hero beats.** Explicitly rank the six tools so that if time runs short, the flip case and the interactive-state case (your two most differentiated moments) are guaranteed shippable and, say, the MCP wrapper or the `update_source` branch is the first thing cut. The plan hints at this, but a stated cut-order removes ambiguity under pressure.
5. **Add one sentence on the messy-Figma reality (small Usefulness insurance).** Acknowledge that orgs without a clean W3C token export need a one-time normalization step, and that the target buyer (the pipeline owner) is exactly who can provide it — closing the only real-world adoption gap a judge might poke at.
