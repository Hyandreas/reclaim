# Judge 1 — Round 1 Review: Driftline

## Total: 13.0 / 15

Driftline is a token-vs-code semantic diff engine that classifies each discrepancy into a closed 4-verdict taxonomy (accidental regression / intentional redesign / platform constraint / orphaned token), grounds the classification in three deterministic signals (migration ratio, commit corroboration, recurrence), and — its central differentiator — decides *which side to fix*, drafting a token-change proposal for the designer when the evidence shows code moved on purpose. It is exposed as an MCP server that Cursor calls natively. This is a strong, unusually self-aware, technically-grounded proposal with minor gaps on each axis.

---

## Novelty: 4.0 / 5

The proposal is refreshingly honest that its spine — a token-vs-code diff with a redesign/regression/platform-constraint taxonomy — is close to an example direction the brief itself hands you, and it correctly declines to claim that restating the brief is the novelty. Three genuine ideas sit on top of that spine, and they are real, not cosmetic. First, **bidirectional reconciliation** ("Driftline will tell the designer they're wrong, not just the engineer") is the strongest and most memorable: every obvious build of this brief assumes code conforms to the token, and flipping that when the evidence says code is the new intent is exactly the concrete mechanism the brief's "keeping designers and engineers aligned" clause asks for. Second, the **verdict-flip on evidence, live on screen** is a strong demo device and a meaningful conceptual distinction between classification (sorting into buckets) and reasoning (changing its mind when signals disagree with the naive read). Third, **recurrence as "the codebase voting for a token that doesn't exist yet"** — turning `Promote to token` from a manual button into a volunteered recommendation — is a clever reframe. What holds this at 4.0 rather than higher is that the foundational concept remains the brief's own example direction, well-executed; none of the three additions is a category-defining reframe of the problem. The "taste" question the brief poses ("enough taste to know when something is wrong") is answered by reducing taste to evidence-counting — defensible and testable, but not surprising.

## Implementation Plan: 4.5 / 5

This is the standout axis. The stack is concrete, real, and appropriate end-to-end (Vite/React/Tailwind, PostCSS + `ts-morph` for real AST-based className/inline-style extraction with exact `file:line:col` spans, `culori` for CIELAB ΔE and WCAG contrast, `better-sqlite3`, `@anthropic-ai/sdk`, `@modelcontextprotocol/sdk`), and every library named actually does what the proposal needs. Critically, I verified the specific Anthropic API claims against the current SDK reference and **all four are correct**: `claude-opus-4-8` is the right current model ID; `thinking: {type: "adaptive"}` is exactly right for it (an `enabled`/`budget_tokens` config would return a 400); structured outputs via `output_config.format` with a `json_schema` is the correct, current path (the older `output_format` is deprecated); and `strict: true` is correctly placed on the tool definition. The proposal's line that this was "verified against the current SDK reference, not assumed" is substantiated — a rare and strong credibility signal. The architecture is genuinely well-risk-managed: Layer 1 (detection, diff, all three signals, WCAG math) is pure code with zero LLM, so the money numbers are reproducible; the LLM touches only classification and short fix-copy, both schema-constrained; fixes are constrained to the scanner's verified byte range so a hallucinated patch is "structurally impossible"; and Day 1 alone produces a demoable deterministic drift list as a fallback. The Tailwind-resolution algorithm — the fiddliest real step — is fully specified with a midday Day-1 gate check, and dynamic classNames are explicitly declared out of scope rather than discovered live. What keeps this at 4.5 rather than 5: the total surface area (six tools + deterministic engine + a seeded repo with realistic git history + bidirectional fix generation + one dashboard + MCP wrapper + audit trail) is a lot for ~2 days even with disciplined scoping, and the `ts-morph`/arbitrary-value-resolution step carries real residual schedule risk — the midday gate detects failure but the stated recovery path is thin, and if that step slips, much of the differentiated value collapses back to the deterministic list.

## Usefulness / Relevance: 4.5 / 5

This is the most literal and complete reading of the track statement I could ask for. The brief names "visual **and interactive** surface" — Driftline runs interaction-state tokens (`hover:`/`focus:`/`focus-visible:`/`disabled:`) through WCAG 2.1 SC 1.4.11 non-text contrast as a first-class check, not a bolt-on. The brief says "keeping designers and engineers aligned" — the bidirectional token-change proposal is the concrete artifact that pulls the designer into the loop, making the alignment claim literal instead of aspirational. The buyer is specific and the problem is framed as budgeted (WCAG/ADA legal exposure converts "aesthetics" into risk), and the "CI gate, not a workflow change" positioning is the most "ships Monday morning" interpretation available. The Cursor/MCP integration honors the sponsor substantively — the same six tools drive both the dashboard and the in-editor experience — rather than as a token gesture. The close-out metric ("14 surfaced, 3 auto-fixed forward, 1 flipped and fixed backward, ~40 min saved on one PR") is concrete and demo-legible. What holds this at 4.5: the "interactive surface" coverage — a named half of the track statement — is deliberately scoped to a single rule (non-text contrast on four pseudo-class variants). The proposal is honest about this ("not a general interaction-state framework"), but it means the interactive half is demonstrated rather than deeply delivered.

## Concrete Strengths

- **Verified API grounding.** The Opus 4.8 model ID, adaptive-thinking config, `output_config.format` structured outputs, and `strict: true` tool schemas are all correct against the current reference. This is the kind of detail most proposals get wrong, and it materially raises confidence that the team can ship.
- **Deterministic core as both fallback and reproducibility guarantee.** Zero-LLM detection/diff/signals means the demo's numbers are pinned to the second, and Day 1 ends with a demoable artifact even if the LLM layer slips.
- **Bidirectional fix is a genuine, memorable insight** and is the concrete mechanism answering the brief's "alignment" clause — not a restatement of it.
- **Risk section is real, not decorative:** midday Day-1 gate on the hardest step, planted marquee bugs with pinned fallback output, a live `git log` cut to prove the reasoning isn't hardcoded, and an explicit `needs-manual-review` bucket for dynamic classNames.
- **Scope discipline** (one token format, one code target, no CV/pixel-diff, one dashboard screen with a single reused diff-card component) is exactly calibrated to a 2-day budget.

## Concrete Weaknesses / Gaps

- **Novelty is capped by the brief's own example.** The core is the diff-and-classify direction the brief describes; the differentiation lives in execution and one conceptual extension. Nothing reframes the problem itself.
- **Schedule risk concentrates in one step.** `ts-morph` extraction plus Tailwind arbitrary-value resolution via ΔE is the value-critical, time-hungry part. The midday gate flags failure but the recovery path if it fails is under-specified — and much of the differentiated demo depends on this step succeeding.
- **The verdict-flip depends on the LLM reliably overturning Layer 1.** It's the single most important demo beat, yet it's the one beat routed through the model's judgment rather than deterministic logic. Pinned fallback output covers a scripted take but not a judge who "points at any of the ~14 findings live."
- **Interactive-surface coverage is one rule.** For a named half of the track statement, a single contrast check reads as a proof-of-concept.
- **The designer-facing artifact stays inside Driftline.** The token-change proposal is rendered as a card/markdown snippet; it never actually reaches where a designer works, which slightly softens the "pulls the designer into the loop" claim.

## Prioritized Fixes (to raise the score)

1. **(Novelty, biggest lever) Elevate one idea from "feature" to "identity."** Make the recurrence-driven "the codebase is teaching the design system a token it doesn't have yet" the headline — a system that *proposes new semantic tokens from evidence* is more original than diff-and-classify, and you already compute the signal. Pair it with the bidirectional fix as the product's spine, and demote the 4-way taxonomy to table stakes. This reframes Driftline from "the brief's example, done well" toward a distinct thesis.
2. **(Implementation) Make the money beat deterministic.** When migration ratio ≥ 80% AND recurrence ≥ 3 (your stated thresholds), compute the "Intentional Redesign" verdict in code and use the LLM only to write the rationale sentence. This removes the last live-AI risk from the verdict-flip and makes it 100% reproducible when a judge points at any finding.
3. **(Implementation) Specify the Day-1 gate's failure branch concretely.** State the exact fallback if arbitrary-value resolution isn't working by midday (e.g., ship CSS-custom-property + Tailwind-palette-class resolution only, drop `bg-[#hex]` arbitrary values, keep the deterministic list + classification intact). A named degraded path protects the demo.
4. **(Usefulness) Add one more interactive check.** A second interaction-state rule (e.g., focus-ring visibility/offset, or hover-state contrast) makes the "interactive surface" half feel delivered rather than demonstrated — cheap, since the pipeline already tags interactive variants.
5. **(Usefulness) Land the designer artifact where designers live.** Render the token-change proposal in a paste-ready Figma-comment or Linear/GitHub-issue format and show that in the 0:17–0:32 beat, turning "we generate a card" into "the designer receives it in their tool."
